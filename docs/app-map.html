Brand Entity & Product Offers — Enterprise wiring gaps

This is the section flagged for wiring into the enterprise tech stack. Backend (tRPC + MySQL) largely exists; the UI is mostly Zustand/localStorage and several powerful server capabilities have no UI caller. There are also 3–4 parallel brand/product models that don't share one source of truth.

The 4 parallel brand/product models (need unifying)

  • Zustand Brand — avatar, nested DigitalProduct[], revenue goals (localStorage). Drives Brands.tsx, Writer pickers, Financial.
  • brands table — name/niche/website/color only. Create-synced from Brands.tsx.
  • brand_contexts table — voice/tone/audience/keywords (BrandVoice.tsx). Optional brandId link unused.
  • Workspace Dexie brands — 10 internal brands w/ "backend offer" + monthly goal for agent matching (workspace/intel.ts).

Recommended wiring sequence (to enterprise stack)

  • Pick one canonical model: extend brands to embed offer + funnel + avatar, or formalize the link brand_contexts.brandId → brands.id.
  • Wire Brands.tsx to data.brands.update/delete + data.products.create/list/delete (stubs already imported).
  • Add a Funnels UI on Brands/Financial that calls the orphaned funnels.* router.
  • Add an "Auto-research brand" button → brandContext.autoResearch.
  • Pass brandContextId + productIds from Pipeline & Social into bridges.*/social.generate so output is brand- and offer-aware.
  • Fix hydration to merge products + brand_contexts into the Zustand brand engine (no more divergent duplicates).
